# Minecraft 指令可视化编排器设计稿

## 1. 产品定位与设计原则

### 产品定位

这个工具不是“把指令翻译成中文”的教学器，也不是“普通表单式命令生成器”。
它应该是一个面向 Minecraft 玩家、基于原生命令体系的低代码图编辑环境：

- 用蓝图式节点连接表达执行顺序、条件关系和逻辑结构
- 用 ComfyUI 式参数面板暴露命令参数、默认值和高级配置
- 用编译预览把图最终落回用户熟悉的 Minecraft 指令或函数输出

用户仍然接触 `give`、`summon`、`execute` 这些原生命令名词，但不再被迫在游戏里用难用的文本框硬写复杂内容。

### 核心价值

这个工具应该稳定兑现 3 个价值：

1. 更容易编辑
替代游戏内文本框，降低长命令、复杂参数和命令方块配置的编辑成本。

2. 更容易发现
让用户直接看到命令支持哪些参数、参数怎么填、哪些是高级用法，而不是靠记忆或查 wiki。

3. 更容易理解
让执行顺序、条件、依赖关系、组合结构在画布上可视化，而不是埋在长指令和多命令方块链里。

### 设计原则

后续所有产品决策都应遵守这 6 条：

1. 原生命令名词保留
不回避 `give`、`summon`、`execute` 等术语，因为目标用户能理解这些词。

2. 节点图是主交互，不是装饰
图必须承担逻辑组织职责，而不是给表单加一层外壳。

3. 参数必须显式可探索
用户不需要背语法，也不应该靠隐藏规则猜支持什么参数。

4. 复杂性渐进展开
默认先展示常用参数和常见连接，高级能力按需展开，避免一上来把全部细节砸给用户。

5. 图上先看懂逻辑，侧栏再精修参数
画布负责结构理解，配置面板负责细节编辑，两者职责清晰。

6. 输出永远可追溯
用户必须能清楚知道“这张图最后生成了什么命令/函数”，不能让图和结果脱节。

### 对当前工具的直接启示

按这个方向看，当前最需要避免的偏差是：

- 不能把产品做成“技术上是节点，体验上还是表单”
- 不能把节点库做成“程序员视角的抽象分类”
- 不能只强调能编译，而忽略“为什么这样连、为什么这样配”

## 2. 信息架构与页面结构

### 信息架构

建议将当前工具的信息架构收敛成 4 个一级区域：

1. 首页
职责：说明这是什么工具、解决什么问题、引导进入编辑器。

2. 编辑器工作台
职责：核心画布工作区，完成节点连接、参数编辑、编译预览、导出。

3. 资源与模板
职责：沉淀常见工作流、示例图、可复用节点组合。
说明：这一块可以先保留轻量入口，不必在当前版本做重。

4. 文档与帮助
职责：解释节点分类、参数含义、连线规则、常见工作流和导出方式。

产品的真正中心必须是“编辑器工作台”，不是首页，也不是文档页。
首页只是入口，文档只是辅助，价值必须在工作台里直接成立。

### 首页结构

首页不应该继续像“功能清单页”，而应该像“工作台入口页”。

建议首页只承担 3 件事：

1. 解释产品一句话定位
例如：
“像蓝图一样连接 Minecraft 指令逻辑，像 ComfyUI 一样直观编辑参数与结果。”

2. 强调 3 个核心价值
编辑更舒服
参数更可见
逻辑更直观

3. 给出明确入口
进入编辑器
查看示例工作流
查看支持的节点类型

首页不建议再把太多技术项并列展示，例如 PWA、Web、节点数、版本号这类信息不应抢主叙事。它们可以保留，但必须降权。

### 编辑器工作台结构

建议工作台固定成“左中右下”四区模型，这很适合“蓝图骨架 + ComfyUI 参数体验”。

1. 左侧：节点库
职责：按功能分类提供可拖拽节点。
要求：分类清晰，搜索快速，节点说明简洁。

2. 中间：主画布
职责：承载逻辑骨架，展示执行流、分支、连接关系。
要求：主流程清晰，节点间关系一眼可见，空状态有引导。

3. 右侧：参数与节点详情面板
职责：编辑当前选中节点的参数、可选项、默认值和高级配置。
要求：像 ComfyUI 一样稳定、直接、连续，不让用户频繁切上下文。

4. 下方或右下扩展区：编译结果与诊断
职责：展示生成结果、错误、警告、缺失参数、无法连线原因。
要求：结果与当前图可追溯，不只是单纯输出文本。

### 编辑器内的用户主流程

围绕当前产品目标，主流程固定为：

1. 从节点库添加节点
2. 在画布中连接逻辑
3. 选中节点配置参数
4. 实时或主动触发编译检查
5. 查看结果与问题
6. 复制或导出

后面所有设计都应该服务于把这 6 步做短、做清楚、做稳定。

### 当前结构存在的问题

结合现有实现，当前结构上主要有这些不合理点：

1. 首页与工作台的价值衔接还不够强

2. 工作台分区已经有雏形，但职责还不够纯

3. 编译结果区更像附加面板，不像工作台的一等反馈区

4. 节点库和配置区还更偏实现结构，不够偏用户任务结构

### 建议的页面职责边界

为了后续不发散，建议直接定这几个边界：

1. 首页不承担教学职责
首页只负责定位和导流，不负责解释全部节点逻辑。

2. 画布不承担细节编辑职责
画布重点是结构，不要把过多细节塞回节点卡片内部。

3. 配置面板不承担全局流程解释职责
它负责当前节点，而不是替代工作流说明。

4. 结果区不只显示最终文本
它还要承担编译诊断、连接问题、缺参提醒和结果追踪。

## 3. 节点系统与分类策略

### 节点系统的基本原则

节点系统遵循这 5 条原则：

1. 节点首先表达 Minecraft 语义，不先表达技术抽象
用户先看到的是 `give`、`summon`、`execute`、`fill`，而不是“动作节点”“资源节点”这种过于中间层的命名。

2. 节点分类按功能场景组织，不按底层实现组织
也就是节点库可以分类，但分类应该帮助玩家找命令，而不是暴露编译器内部结构。

3. 常用节点必须一眼能找到
高频节点不能埋在多层分类里，否则低代码优势会被搜索成本抵消。

4. 节点要有“骨架层”和“细节层”
骨架层在画布上表达这是什么命令、它与谁相连；细节层放到右侧参数面板里，不要把节点卡片做得过重。

5. 节点之间的连接必须有明确语义
不是“都能连”，而是让用户理解执行流、数据输入、条件依赖分别是什么。

### 建议的节点分类

建议节点库先分成下面几组：

1. 基础命令
`give`、`summon`、`tp`、`kill`、`effect`、`clear`

2. 方块与世界
`setblock`、`fill`、`clone`、`time`、`weather`、`gamerule`

3. 文本与反馈
`tellraw`、`title`、`playsound`、`particle`

4. 条件与执行
`execute`、条件判断、选择实体、位置上下文、条件分支

5. 计分板与数据
`scoreboard` 相关、storage 相关、变量读写

6. 流程控制
顺序执行、条件分支、循环、延时、计划执行

7. 常量与输入
数字、字符串、坐标、选择器、资源 ID

命令节点和流程节点要并存，但命令节点始终是主角，流程节点是为了让命令逻辑更清楚，而不是反过来把命令变成流程图配角。

### 节点库的层级策略

节点库采用“两层，不超过两层”的结构：

1. 第一层是功能分类
比如“基础命令”“方块与世界”“条件与执行”

2. 第二层是具体节点
比如 `give`、`summon`、`execute`

不要再往下做三级目录。
如果节点多，优先靠搜索、置顶常用和最近使用解决，而不是继续加层级。

### 哪些节点应该是一等公民

当前版本要突出的一等节点，建议是：

1. `give`
2. `summon`
3. `tp`
4. `setblock`
5. `fill`
6. `effect`
7. `execute`

这些节点既是玩家熟悉的核心命令，也是最能体现“参数复杂、逻辑容易混乱、适合可视化”的命令。

第二梯队再考虑：

1. `tellraw`
2. `title`
3. `playsound`
4. `particle`
5. `scoreboard`
6. `schedule`

第三梯队再逐步扩展复杂数据节点和高级逻辑节点。

### 节点本体应该长什么样

每个节点卡片只保留这些核心信息：

1. 节点名称
如 `give`

2. 节点简短描述
一句话说明用途

3. 输入/输出连接点
清楚区分执行和数据

4. 关键摘要
只显示 1 到 2 个最重要参数摘要
例如：
`give`
`target: @p`
`item: minecraft:diamond`

不要把完整表单塞进节点本体，否则画布会很快失控。

### 节点分类上当前工具的问题

从现状看，当前分类思路已经有基础，但还存在几个问题：

1. 分类名称有些偏内部结构，不够像玩家任务入口

2. 节点量感强，但优先级不够明显

3. 命令节点与流程节点的关系还不够清楚

4. 常量/辅助输入节点的产品地位还没完全建立

### 建议补强的节点体验

建议节点库后续加入这几种增强机制：

1. 常用节点置顶
让 `give`、`summon`、`execute` 这类节点固定在顶部可见。

2. 最近使用
降低重复搜索成本。

3. 分类内简短说明
比如“条件与执行：控制命令在什么条件下运行”。

4. 节点搜索支持命令名和用途词
既能搜 `give`，也能搜“给予”“传送”“召唤”。

5. 示例工作流入口
让用户从“给玩家物品”“召唤实体并播放效果”这类常见起点进入。

### 这一节的结论

节点系统不应该追求“万物皆抽象”，而应该追求：

- 用原生命令名词建立熟悉感
- 用功能分类降低查找成本
- 用流程节点建立逻辑表达能力
- 用轻量节点本体维持画布可读性
- 用参数面板承载复杂细节

## 4. 画布交互与连线规则

### 画布的职责

画布不是展示所有细节的地方，而是用户理解逻辑结构的主界面。
它应该优先回答 4 个问题：

1. 这个流程从哪里开始
2. 当前命令按什么顺序执行
3. 哪些命令受条件控制
4. 哪些参数或结果来自其他节点

如果用户不能在几秒内回答这 4 个问题，说明画布设计仍然不够成立。

### 两类连线必须明确区分

建议将连线明确分成两类：

1. 执行流
表示“先做什么，再做什么”，是蓝图式逻辑骨架的核心。

2. 数据流
表示“这个节点需要的参数来自哪里”，是 ComfyUI 式参数连接能力的核心。

这两类线必须在视觉上稳定区分，不能只靠用户猜。
至少要在颜色、线型、pin 形状、悬停说明中占两项以上差异。

### 连线规则的基本约束

建议连线规则遵循下面原则：

1. 执行流只能连执行流
不能把数据 pin 当作执行顺序连接。

2. 数据流只能在兼容类型之间连接
例如坐标到坐标、数字到数字、实体到目标类输入。

3. 不兼容连接要即时阻止，并解释原因
不要只是不让连上，要告诉用户“类型不匹配”或“该输入不接受这种数据”。

4. 一个输入是否允许多个来源必须明确
如果某参数只接受一个来源，就要在视觉和交互上说明。

5. 必填输入缺失必须在节点层面暴露
不能等用户编译后才发现。

### 建议的 pin 语义

建议 pin 采用固定语义，减少学习成本：

1. 左侧输入，右侧输出
保持稳定，不在不同节点上反复改变。

2. 圆形 pin 表示执行流
强调流程控制属性。

3. 方形或其他形状 pin 表示数据流
强调参数或值传递属性。

4. 不同数据类型采用稳定颜色
例如实体、坐标、数字、字符串、NBT、资源 ID 各有固定色。

这样用户在不读太多文字时，也能逐渐形成视觉记忆。

### 空画布引导

空画布状态不能只提示“拖拽节点开始”。
它至少应该提供：

1. 一个推荐起点
例如“先拖入 `execute` 或 `give` 节点开始”。

2. 一个示例工作流入口
例如“加载一个给物品示例”。

3. 一段极短的连线说明
例如“圆点控制执行顺序，方点传递参数”。

这会明显降低首次使用门槛。

### 选中、悬停与局部聚焦

为了让复杂图也能看懂，建议补强这几种交互：

1. 选中节点时，高亮它的上下游连接
方便用户看清该节点前后关系。

2. 悬停 pin 时，高亮可连接目标
降低试错成本。

3. 选中连线时，显示这条线的语义说明
比如“执行流”“目标实体”“坐标输入”。

4. 节点存在缺参或错误时，在节点层直接标识
不要把问题全推到编译结果区。

### 复杂逻辑的可读性策略

如果参考 UE 蓝图，复杂流程不可避免，因此需要预先设计可读性机制：

1. 主流程应鼓励横向或纵向统一方向
减少回折和交叉线。

2. 条件分支节点必须有明显的真假出口
不能让用户靠线条猜哪条是成功、哪条是失败。

3. 延时、循环、条件等流程节点应有更强视觉特征
使其区别于普通命令节点。

4. 允许后续加入节点分组、注释框或区域标记
为复杂工作流留扩展空间，但当前版本不必一次做完。

### 当前工具在画布层面的主要问题

结合现状，当前主要差距在于：

1. 连线类型虽然有数据类型概念，但对用户的显式解释仍偏弱

2. 空状态引导不够强

3. 节点与连线的错误反馈更多停留在底层校验，还没有产品化表达

4. 画布可读性策略还偏基础，缺少对复杂流程的聚焦辅助

### 这一节的结论

画布交互的目标不是“能连起来”，而是让用户：

- 看懂执行顺序
- 看懂数据来源
- 看懂哪里出错
- 看懂最终为什么会生成这段命令逻辑

只要这 4 件事没成立，低代码体验就还不算真正成立。

## 5. 参数面板设计

### 参数面板的职责

参数面板不是简单的“表单区”，而是节点工作流中的细节编辑中心。
如果画布负责让用户看懂结构，那么参数面板就负责让用户高效、连续地完成参数配置。

它应该重点解决 4 件事：

1. 这个节点能配置什么
2. 哪些参数是常用的，哪些是进阶的
3. 当前参数是否缺失或不合理
4. 修改参数后会对结果产生什么影响

### 参数面板的基本原则

建议参数面板遵循这 6 条原则：

1. 默认展示常用参数
先把用户最常改、最影响结果的参数露出来。

2. 高级参数按需展开
不要把所有配置一次性铺开，避免面板噪音过大。

3. 参数名称保留 Minecraft 语义，但说明要更友好
例如保留 `target`、`item`、`nbt` 等原生概念，同时配上简短说明。

4. 默认值必须合理
让用户拖进节点后即使不全改，也能快速得到一个可理解的结果。

5. 参数修改应尽量即时反馈
减少“改完还不知道影响了什么”的断层。

6. 数据输入与手动输入要能共存
如果某个参数既可以手填，也可以接线输入，面板要能清楚表达当前来源。

### 建议的面板结构

建议右侧参数面板固定为 5 个区块：

1. 节点头部
显示节点名、简短说明、当前状态。
例如：`give`、用途说明、是否缺参、是否有警告。

2. 常用参数区
展示高频核心参数。
例如 `give` 节点优先展示目标、物品、数量。

3. 高级参数区
折叠显示不常用但重要的能力。
例如 NBT、自定义名称、特殊标志、执行上下文等。

4. 输入来源区
显示某些参数当前是“手动填写”还是“由连线提供”。

5. 节点诊断区
集中展示当前节点缺参、类型不匹配、逻辑警告等问题。

这样面板既能维持 ComfyUI 式的稳定工作流感，也不会退化成无结构长表单。

### 参数分层策略

建议每个节点的参数按 3 层处理：

1. 核心参数
这是用户几乎必填、几乎每次都会关注的参数。

2. 可选参数
能提升表达力，但不是每次都必须配置。

3. 高级参数
适合有经验用户或复杂命令场景，需要折叠或按需显示。

以 `summon` 为例：

1. 核心参数
实体类型、位置

2. 可选参数
朝向、名称、目标选择器

3. 高级参数
NBT、复杂行为属性、附加状态

这样既保留命令原貌，也不会让每个节点一打开就变成语法大全。

### 参数控件建议

为了提高可编辑性，建议不同类型参数采用稳定控件：

1. 选择器参数
提供常见选择器快捷项，同时允许手写自定义内容。

2. 坐标参数
支持相对坐标占位提示，如 `~ ~ ~`，并可扩展为三段式输入。

3. 资源 ID 参数
保留文本输入，但应支持示例、自动补全或候选提示。

4. 数值参数
提供数值框、上下限和默认值。

5. 布尔或枚举参数
优先使用切换、下拉、单选，不要让用户手写。

6. NBT 参数
不建议一开始全面展开成复杂结构编辑器，但至少要有单独高级区域、示例和校验提示。

### 手填与连线共存的规则

如果参考 ComfyUI，参数面板必须很好处理“手动值”和“连线值”的共存关系。

建议规则如下：

1. 参数未接线时，可直接手填

2. 参数已接线时，默认显示为“由节点输入提供”

3. 用户仍应能看到该参数原本是什么类型、期望什么值

4. 如果允许手填值作为默认回退，需要明确标识优先级

5. 如果接线后手填值失效，需要在 UI 上清楚提示

这部分如果处理不好，用户会经常陷入“为什么我改了这里却没生效”的困惑。

### 参数说明与示例

当前工具如果想真正解决“参数和高级用法不直观”的问题，参数面板必须承担解释职责。

建议每个重要参数具备以下辅助信息中的至少一项：

1. 一句话说明
说明它控制什么。

2. 占位示例
例如 `minecraft:diamond`、`~ ~ ~`、`@p`。

3. 常见用法提示
例如“通常用于最近玩家”或“可连接坐标节点覆盖手填值”。

4. 错误提示
例如“数量必须大于 0”或“该输入需要实体选择器”。

注意这里不是要做百科式文档，而是做“刚好够用”的上下文帮助。

### 当前工具在参数面板上的问题

结合现状，当前主要问题是：

1. 面板已经能编辑字段，但还偏“字段渲染器”

2. 参数分层不够明显
常用参数和高级参数还没有被拉开层次。

3. 连线输入与手填输入的关系还没有被产品化表达

4. 参数说明、示例、默认值策略还不够系统

5. 节点级错误提示还没有形成稳定区域

### 这一节的结论

参数面板不应只是“把命令拆成输入框”，而应成为：

- 参数探索中心
- 节点诊断中心
- 手填与连线输入的协调中心
- 结果变化的可追踪编辑中心

只有做到这一点，ComfyUI 式参数体验才算真正成立。

## 6. 资源库与输入节点设计

### 为什么需要资源库

如果 `give`、`setblock`、`summon`、`playsound` 这类节点仍然主要依赖用户手填 `minecraft:*` 形式的资源 ID，那么这个工具虽然有了节点画布，但在参数体验上仍然保留了原版命令编辑最不友好的部分。

因此，资源库不是一个可有可无的附加功能，而是“参数可发现性”和“低代码输入体验”的关键组成部分。

它要解决 4 个问题：

1. 降低手写资源 ID 的成本
2. 降低拼写错误和命名记忆负担
3. 让资源类参数更可探索
4. 让资源来源在图上可以显式表达和复用

### 资源库的定位

资源库不应只理解为“物品库”，而应是一套统一的资源输入体系。

首批建议纳入：

1. 物品
2. 方块
3. 实体类型
4. 状态效果
5. 粒子
6. 音效

这样它不只服务 `give`，还可以同时服务 `summon`、`setblock`、`fill`、`effect`、`particle`、`playsound` 等节点。

### 资源输入的双通道策略

不建议把所有资源参数都强制节点化，也不建议继续只保留手填文本。

建议采用“双通道”：

1. 面板选择
用户可以直接在参数面板里搜索并选择资源。

2. 输入节点
用户也可以拖入“资源节点”并通过连线提供参数。

这个策略兼顾两类场景：

1. 简单场景优先效率
例如给玩家一个钻石，直接在 `give` 节点参数面板里选物品即可。

2. 复杂场景优先结构表达与复用
例如多个节点都依赖同一个物品、方块或实体类型时，可以通过资源节点统一供给。

### 资源节点的产品价值

资源节点不是为了“让画布更热闹”，而是为了让图中的数据来源更明确。

它们在工作流里主要承担 3 个作用：

1. 复用
同一个资源值可以供多个命令节点使用。

2. 可视化依赖
用户能从图上直接看出某个参数来自哪里。

3. 解耦
后续改动资源值时，不必逐个改多个命令节点。

这与 ComfyUI 中常量输入、模型输入、参数输入节点的价值是一致的。

### 建议的一等资源节点

建议优先提供以下资源输入节点：

1. 物品节点
输出 `item/resource`

2. 方块节点
输出 `block/resource`

3. 实体类型节点
输出 `entityType/resource`

4. 选择器节点
输出 `selector/entity`

5. 坐标节点
输出 `position`

6. 数值节点
输出 `number`

7. 字符串节点
输出 `string`

这里的关键不是节点越多越好，而是先把最常用、最能提升可用性的输入节点做透。

### 资源面板体验建议

如果资源库只是一份长列表，它依然会不好用。

建议资源面板至少具备：

1. 搜索
支持按英文 ID、常见中文用途词搜索。

2. 分类
例如武器、工具、方块、生物、药水效果等。

3. 常用项
优先展示高频资源。

4. 最近使用
减少重复搜索。

5. 示例预览
至少展示资源名和 ID，后续可考虑图标或简短说明。

### 资源节点与参数面板的关系

资源节点和参数面板不是互斥关系，而应该是同一套参数系统的两个入口。

建议规则如下：

1. 默认可在面板中手动选择资源
2. 如果该参数已接入资源节点，面板显示当前来源为“来自连线输入”
3. 面板仍应展示该参数的期望类型与当前值摘要
4. 若用户断开连线，面板可恢复为手动编辑状态
5. 若产品允许“面板默认值 + 连线覆盖”，必须明确覆盖优先级

### 当前工具在这方面的缺口

结合现状，当前主要缺口是：

1. 资源参数大多还是文本输入思路
2. 资源类值缺少统一搜索和选择入口
3. 资源来源在图上还不够显式
4. 可复用输入节点的产品价值还没有被建立起来

### 这一节的结论

资源库应被视为低代码体验的基础设施，而不是补充小功能。

合理的方向不是：

- 只保留手填资源 ID
- 或强制所有资源都必须先拖一个输入节点

而是：

- 面板选择保证简单场景效率
- 输入节点保证复杂场景的结构表达和复用

这样既保留了 Minecraft 原生命令语义，也真正提升了图编辑器的表达力。

## 7. 编译反馈与错误提示

### 编译反馈的定位

编译反馈区不应只是一个“输出文本框”，而应是连接图编辑与最终结果的解释层。

它至少要回答 4 个问题：

1. 当前图能否成功生成结果
2. 生成了什么命令或函数
3. 如果失败，失败在什么节点或哪条连接上
4. 如果成功，是否仍存在潜在问题或不完整配置

### 反馈分层

建议把反馈分成 3 层，而不是只保留“成功/失败”两态：

1. 成功
图已正确生成输出，且没有明显问题。

2. 警告
图可以生成输出，但存在潜在风险或不完整之处。
例如使用了默认值、存在未连接的可选逻辑、某些节点配置可能不符合预期。

3. 错误
图无法正确生成输出，必须修复后才能继续。
例如必填参数缺失、连接类型不匹配、流程断裂、非法组合。

### 错误提示的基本原则

建议错误提示遵循这 5 条原则：

1. 错误必须能定位到具体对象
告诉用户是哪个节点、哪个参数、哪条连接出了问题。

2. 错误描述必须可操作
不要只说“编译失败”，而要说“`give` 节点缺少 item 参数”。

3. 能前置发现的问题就不要延迟到编译后
例如缺参、类型错误、无效连接，应尽量在节点层即时暴露。

4. 错误和警告必须与 Minecraft 心智对齐
提示语言应更贴近命令和命令方块场景，而不是仅从编译器角度叙述。

5. 成功反馈也要有解释性
让用户看懂“为什么现在是对的”，而不是只给一个绿色状态。

### 结果区应该展示什么

建议结果区至少包含 4 类内容：

1. 输出结果
显示最终生成的命令、命令序列或 `mcfunction` 片段。

2. 节点级诊断列表
列出错误、警告和缺失项，并支持点击回到对应节点。

3. 结构摘要
例如主流程长度、关键命令数量、是否包含条件分支、是否有未使用节点。

4. 导出动作
复制、导出、下载。

### 从“结果展示”升级到“结果解释”

如果参考蓝图和 ComfyUI，用户需要的不只是结果本身，而是结果与结构的对应关系。

因此建议增加：

1. 节点到结果的映射
点击节点时，结果区可高亮其对应输出片段。

2. 结果到节点的追溯
点击结果中的某段命令，可反向定位对应节点。

3. 编译前后状态一致性
用户调整节点后，应清楚知道结果已过期、待重新检查还是已同步更新。

### 典型错误类型

建议先明确支持这几类高频错误：

1. 必填参数缺失
例如 `give` 缺少物品，`summon` 缺少实体类型。

2. 数据类型不匹配
例如把数字连接到坐标输入。

3. 流程断裂
例如执行流中断、没有起点或存在悬空分支。

4. 非法组合
例如某些节点只能用于特定执行上下文。

5. 未使用节点
节点存在但没有进入主流程，也没有被其他节点消费。

### 反馈呈现建议

建议采用以下呈现方式：

1. 节点层即时标识
红色错误、黄色警告、灰色未接入等状态直接显示在节点上。

2. 结果区集中汇总
给出完整问题列表与输出内容。

3. 可跳转修复
点击错误项时，自动聚焦到对应节点。

4. 文案简洁直接
避免长篇术语解释，优先给出修复动作。

### 当前工具在这方面的主要问题

结合现状，当前主要差距是：

1. 编译结果更像纯输出面板，解释性不足
2. 错误与节点的映射关系还不够强
3. 缺参、类型问题等还没有形成统一的产品反馈层
4. 成功状态下的结构说明还不足，用户不容易建立信心

### 这一节的结论

编译反馈区不应只是“最后给你一段文本”，而应成为：

- 结果解释中心
- 错误定位中心
- 修复导航中心
- 图与输出之间的追溯中心

只有这样，用户才会觉得这张图不仅能画，而且值得画。

## 8. 新手引导与示例工作流

### 新手引导的定位

虽然目标用户通常懂一些基础命令，但这不意味着他们天然理解“节点化命令编排”。

因此，新手引导的职责不是教用户什么是 `give` 或 `summon`，而是帮助用户建立这套工具特有的工作心智：

1. 节点代表命令或输入来源
2. 线代表执行顺序或数据传递
3. 右侧面板负责配置细节
4. 结果区负责解释最终输出和问题

也就是说，引导的重点不是 Minecraft 基础知识，而是“如何在这个工作台里思考和操作”。

### 首次使用必须完成的 3 个认知建立

第一次进入编辑器时，用户至少要快速理解这 3 件事：

1. 如何把节点放到画布上
拖拽或双击添加节点。

2. 如何连接节点
理解执行流和数据流的区别。

3. 如何得到结果
知道配置节点、查看结果、复制或导出。

如果第一次使用还不能清楚完成这 3 件事，说明引导仍然不够有效。

### 引导方式建议

建议采用“轻引导 + 示例驱动”，而不是大段教学弹窗。

推荐组合是：

1. 空画布提示
用最短文案告诉用户第一步做什么。

2. 首次提示浮层
只在首次进入时出现，讲清节点、连线、参数面板、结果区四个区域。

3. 示例工作流入口
用户可以一键加载一个完整示例，边看边改。

4. 上下文提示
在用户选中节点、悬停连接点、出现错误时给出局部说明。

这比一次性塞给用户整套教程更有效。

### 示例工作流的重要性

如果参考 UE 蓝图和 ComfyUI，示例工作流几乎是产品学习成本控制的核心。

因为很多用户不是通过“看说明”学会的，而是通过：

1. 先打开一个能运行的例子
2. 看懂它由哪些节点组成
3. 改一两个参数试效果
4. 再开始自己搭图

所以示例工作流不是附加内容，而应视为产品主功能的一部分。

### 首批建议提供的示例工作流

建议首批至少内置这些示例：

1. 给玩家物品
最简单的命令场景，用于理解节点添加、参数配置和结果输出。

2. 在指定位置召唤实体
用于理解坐标、实体类型和命令节点配置。

3. 执行条件后给予奖励
用于理解 `execute`、条件分支和执行流。

4. 区域填充方块
用于理解方块资源、坐标输入和批量操作。

5. 播放粒子与音效
用于理解多节点组合和反馈型命令。

这几类示例覆盖了最基础、最有代表性的低代码心智。

### 示例工作流应如何设计

示例不应只是“能运行的一张图”，而应具备教学属性。

建议每个示例都包含：

1. 标题
例如“给最近玩家一把钻石剑”。

2. 一句话说明
说明这个示例解决什么问题。

3. 清晰的节点布局
不要为了展示复杂能力故意堆很多节点。

4. 可直接修改的关键参数
让用户改动后能立即看到变化。

5. 简短说明卡
指出这张图里哪些是执行流、哪些是资源输入。

### 引导的节奏控制

引导必须克制，不能让用户觉得每一步都被打断。

建议节奏如下：

1. 第一次进入
只说明工作台四区和基本交互。

2. 第一次拖节点
提示如何连接。

3. 第一次选中节点
提示右侧面板可编辑参数。

4. 第一次编译失败
提示如何从问题列表跳回节点修复。

这是一种“按需出现”的引导，而不是一次性灌输。

### 当前工具在引导层面的主要问题

结合现状，当前主要问题是：

1. 空状态提示还不够强，难以把用户带入正确起点

2. 示例工作流入口还没有成为主要学习路径

3. 用户第一次使用时，对执行流、数据流、参数面板和结果区的关系还缺少统一解释

4. 学习成本更多依赖用户自己摸索，而不是通过产品主动降低

### 这一节的结论

这类产品的学习方式不应是“读完文档再开始”，而应是：

- 先看一个能运行的图
- 再改一个参数
- 然后理解连接和结构
- 最后开始自己搭建

因此，新手引导和示例工作流必须被当作核心产品能力，而不是补充材料。

## 9. 分阶段实施计划

### 实施目标

设计稿的目标不是一次性做完所有功能，而是先把产品核心体验打磨成立。

因此实施计划应围绕“先建立正确骨架，再逐步增强”的原则推进。

### 第一阶段：修正产品表达与基础可用性

目标：让用户一进入产品，就能理解这是什么、为什么值得用，并且界面文本可读。

建议包含：

1. 修复全局乱码与编码问题
统一源码、文档、界面文案编码。

2. 重写首页定位与核心文案
聚焦“编辑更舒服、参数更可见、逻辑更直观”。

3. 优化空画布提示
增加推荐起点、示例入口、执行流/数据流简述。

4. 统一节点与面板中的基础文案
保证术语稳定一致。

这一阶段不追求功能扩展，先解决“产品有没有正确表达自己”和“用户能不能看懂”。

### 第二阶段：建立成立的图编辑核心体验

目标：让“蓝图式逻辑骨架 + ComfyUI 式参数体验”在主工作流里真正成立。

建议包含：

1. 重组节点库分类
按功能场景而不是底层实现组织节点。

2. 强化执行流与数据流的视觉区分
完善 pin 语义、线条区分和连接反馈。

3. 优化节点本体信息结构
保留名称、摘要、连接点，减少节点卡片噪音。

4. 重构参数面板
引入常用参数、高级参数、输入来源、节点诊断分区。

5. 建立节点级错误与警告状态
把缺参、类型错误等问题前置到节点层。

这一阶段完成后，产品核心交互应已具备说服力。

### 第三阶段：补强资源输入与结果解释能力

目标：让资源探索和编译反馈真正体现低代码价值。

建议包含：

1. 建立资源库
优先支持物品、方块、实体类型、效果、粒子、音效。

2. 增加输入节点体系
至少补齐物品、方块、选择器、坐标、数值、字符串节点。

3. 打通面板选择与连线输入的双通道
明确参数来源和覆盖规则。

4. 重做编译结果区
支持错误、警告、输出结果、节点跳转和结果追溯。

这一阶段的目标是让产品从“可用节点编辑器”升级成“真正有优势的低代码工作台”。

### 第四阶段：补强学习曲线与复用能力

目标：让新用户更容易入门，让中级用户更愿意持续使用。

建议包含：

1. 内置示例工作流
覆盖常见 Minecraft 场景。

2. 增加首次引导与按需提示
降低首次使用门槛。

3. 引入最近使用、常用节点、常用资源
降低重复操作成本。

4. 预留模板与工作流复用能力
为后续资源与模板区做铺垫。

### 实施优先级原则

整个计划建议始终遵守这 4 条优先级原则：

1. 先解决可读性，再扩展功能面

2. 先打磨高频命令节点，再扩展长尾节点

3. 先让核心工作流成立，再补外围能力

4. 先做简单场景高效，再支持复杂场景优雅

### 最终结论

这份设计稿的核心不是追求“功能更多”，而是把产品真正做成：

- 一个能承载 Minecraft 原生命令语义的节点工作台
- 一个让逻辑结构比命令方块更直观的图编辑器
- 一个让参数探索和结果反馈都更友好的低代码环境

如果后续实施过程中始终围绕这个目标收敛，产品方向就不会跑偏。

## 10. 自定义函数节点与封装能力

### 为什么这是必要能力

如果这个工具只支持把所有逻辑平铺在一张画布上，那么当用户开始处理数据包、多段命令链、可复用逻辑片段或较大的工作流时，画布很快就会失去可读性。

因此，“自定义函数节点 / 子图封装”不是锦上添花，而是保证产品能够承载中大型逻辑的结构能力。

它主要解决 4 个问题：

1. 复用
把已经写好的逻辑重复用于多个地方，而不是反复复制节点。

2. 结构清晰
把细节折叠起来，让主画布保留更高层级的逻辑骨架。

3. 数据包输出
让一组节点逻辑可以自然映射到函数、子函数或可导出的结构单元。

4. 可扩展
允许用户逐步建立自己的逻辑库，而不是永远停留在一次性拼图。

### 参考对象的正确映射

如果参考 UE 蓝图和 ComfyUI，这里的能力更接近：

1. 蓝图中的函数封装、宏、折叠节点
2. 工作流中的可复用子图
3. 把一段复杂逻辑提升为高层节点

所以它的本质不是“用户自己写一段文本函数”，而是：
用户可以把一组现有节点打包成一个具备输入输出接口的新节点。

### 建议的能力分层

为了避免一开始过度设计，建议把封装能力分成 3 层：

1. 第一层：折叠为子图
用户可以把一组选中的节点折叠成一个“封装节点”，用于清理主画布结构。

2. 第二层：命名并定义输入输出
用户可以给这个封装节点命名，并明确暴露哪些执行口、哪些数据输入输出。

3. 第三层：保存为可复用函数节点
用户可以把封装后的节点保存到自己的函数库，在其他工作流里复用。

这样推进符合 YAGNI，也符合产品成长路径。

### 这个能力应如何呈现

建议把封装能力设计成“从画布中生长出来”的能力，而不是额外塞一个独立系统。

推荐流程：

1. 用户在画布中选中一组节点
2. 执行“封装为函数节点”或“折叠为子图”
3. 系统让用户确认名称、描述、输入输出接口
4. 生成一个新的高层节点替代原有节点组
5. 用户可进入该节点内部继续编辑

这比先让用户手工创建一个“空白自定义函数节点”更符合直觉。

### 输入输出接口设计

封装能力成败的关键在于接口设计。

建议每个自定义函数节点至少支持：

1. 执行输入
2. 执行输出
3. 数据输入
4. 数据输出

并且接口必须显式定义，而不是靠内部节点自动猜测到不可理解。

如果接口定义不清，封装就只会制造新的复杂度。

### 与数据包能力的关系

如果后续产品要更好支持数据包，这项能力尤其重要。

因为它天然可以成为：

1. 一个逻辑函数的编辑单元
2. 一个可导出的函数文件映射单元
3. 一个较大工作流中的模块边界

也就是说：
“函数节点”不仅是画布整理工具，也可以成为导出结构的一部分。

### 与资源节点、普通命令节点的关系

自定义函数节点不应替代普通命令节点，而应位于其上一层。

可以理解为：

1. 普通命令节点：表达原生命令
2. 资源节点：表达输入值与资源来源
3. 函数节点：表达更高层级的逻辑模块

这三层组合起来，才能真正接近蓝图式分层结构。

### 当前设计稿里原本缺失的点

现有设计稿已经覆盖了：

1. 节点基础组织
2. 参数体验
3. 资源输入
4. 编译反馈
5. 引导与示例

但确实缺少了“如何控制大型工作流复杂度”这一层。

而自定义函数节点正是这一层的核心能力。

### 建议纳入实施计划的方式

建议把它放入实施计划中的第三、第四阶段之间：

1. 在核心图编辑体验成立后引入“折叠为子图”
2. 资源输入与结果解释能力补强后，再引入“可复用函数节点库”

原因是：

1. 太早做会把基础节点体系和接口语义一起复杂化
2. 太晚做会导致用户一旦开始搭复杂逻辑，画布马上失控

### 最小可行版本建议

这个能力的 MVP 不应一开始就做成完整脚本系统。

建议最小版本只做：

1. 选中多个节点
2. 封装成一个新节点
3. 手动定义输入输出
4. 支持进入内部编辑

先解决“画布过于庞杂”的核心问题，再考虑：

1. 函数节点库
2. 全局复用
3. 跨项目导入导出
4. 更强的版本管理

### 这一节的结论

如果产品要真正支持：

- 更复杂的逻辑
- 更清晰的主画布
- 数据包导出
- 用户自己的逻辑资产积累

那么自定义函数节点与封装能力就是必要项，不是可选项。

它应被视为这个工具从“节点编辑器”走向“可扩展低代码工作台”的关键跃迁能力。

## 11. 存储、项目结构与数据模型

### 为什么这一层必须设计

如果前面只讨论了节点、画布、资源、函数封装和编译反馈，但没有定义这些内容如何保存、组织和映射，那么后续一旦进入实现阶段，就会很容易出现结构混乱、状态难以维护、导出能力难以扩展的问题。

因此，存储与项目结构不是纯工程问题，它直接决定了这个工具能否从“单次使用的编辑器”成长为“可持续积累工作流资产的工作台”。

### 项目模型的基本单位

建议把用户的工作成果组织成“项目”而不是单张图。

每个项目至少应包含：

1. 基础元信息
项目名称、描述、版本、创建时间、更新时间。

2. 主工作流
当前用户主要编辑的画布。

3. 子图 / 自定义函数
用户封装出来的可复用逻辑单元。

4. 资源配置
项目内使用到的资源、常用项、最近使用记录或局部资源映射。

5. 导出配置
例如导出为单条命令、函数、数据包时所需的命名空间和选项。

这样可以避免把所有内容都塞进一份平铺 JSON 里，后续难以演进。

### 建议的项目结构

概念上，建议采用如下层次：

1. 项目
最大的工作单元。

2. 工作流
一个可独立编辑和编译的节点图。

3. 节点
命令节点、流程节点、资源节点、函数节点等。

4. 接口定义
描述工作流或函数节点暴露出的输入输出。

5. 资源条目
描述物品、方块、实体等可选择或可复用输入。

其中最关键的是：
“主画布”和“封装函数内部图”在底层都应视为工作流，只是用途不同。

这样自定义函数节点的实现才会自然。

### 工作流作为统一抽象

建议将以下对象都统一视为“工作流”：

1. 主编辑画布
2. 封装后的子图
3. 可导出的函数逻辑

区别只在于它们的元信息和暴露接口不同。

这样有几个好处：

1. 编辑器模型统一
主图和子图都能复用相似的编辑逻辑。

2. 函数节点天然可映射到内部工作流

3. 后续数据包导出更自然
因为每个函数本来就是一个独立工作流单元。

### 节点数据模型建议

建议节点数据至少区分以下层：

1. 通用字段
`id`、类型、位置、标签、描述、状态。

2. 端口定义
执行口、数据输入口、数据输出口。

3. 配置数据
该节点自身的参数值。

4. 来源信息
哪些值来自手填，哪些来自连线，哪些来自默认值。

5. 诊断信息
错误、警告、缺参、未接入状态等。

6. 元能力信息
例如该节点是否为封装函数节点、是否可进入内部编辑、映射到哪个工作流。

这会比“仅存一坨 config”更能支持后续演进。

### 函数节点的数据模型建议

封装函数节点建议额外包含：

1. 关联工作流 ID
指向其内部子图。

2. 接口定义
明确有哪些输入输出。

3. 版本或快照信息
便于后续处理引用更新。

4. 是否为项目内私有
还是已加入全局可复用库。

这部分如果不提前设计，后续一旦要支持复用、导入导出或升级函数节点，代价会很大。

### 资源模型建议

资源不应只是一段字符串，而建议至少具备：

1. 资源 ID
例如 `minecraft:diamond_sword`

2. 资源类型
物品、方块、实体、粒子、音效等。

3. 显示名称
便于面板和搜索展示。

4. 分类信息
便于筛选与浏览。

5. 可选附加元数据
例如别名、常用标签、版本兼容信息。

这样后续资源库的搜索、分类、常用项才能成立。

### 导出结构建议

既然产品未来要支持数据包和函数导出，那么导出模型不应只在最后一刻拼字符串。

建议导出层至少区分：

1. 视图层工作流
用户看到和编辑的图结构。

2. 逻辑层中间表示
编译器更容易处理的统一结构。

3. 输出层结果
命令文本、函数文本、数据包目录结构。

这样可以避免 UI 数据模型和编译器输出模型强耦合。

### 本地存储与项目持久化建议

在产品层面，建议至少支持：

1. 自动保存当前项目
降低用户丢失工作流的风险。

2. 项目列表
让用户能管理多个工作流项目。

3. 项目导入导出
便于分享、备份和跨设备迁移。

4. 示例项目与用户项目分离
避免用户误改内置示例。

如果只保存“当前画布状态”，这个工具很难成长成真正的工作台。

### 当前设计需要补强的边界

基于前面的设计稿，这一层需要明确的边界是：

1. 主画布与子图在底层统一为工作流

2. 节点配置数据与连接来源信息要分开

3. 资源要有独立模型，而不是散落在各节点文本值中

4. 导出能力应依赖稳定中间表示，而不是直接从 UI 拼接

### 最小可行版本建议

这一层的 MVP 不必一次做成复杂项目管理系统。

建议最小版本先做到：

1. 一个项目包含主工作流和多个子图

2. 节点、连接、接口、资源引用有稳定 JSON 结构

3. 支持本地自动保存与手动导出项目文件

4. 编译器从统一工作流结构读取，而不是直接从 UI 临时状态读取

先把模型打稳，比先加更多功能更重要。

### 这一节的结论

如果没有项目结构与统一数据模型，前面的：

- 资源输入
- 自定义函数节点
- 数据包导出
- 项目复用

都会在实现时互相缠绕。

所以这一层虽然看起来偏底层，但其实是整个产品后续可扩展性的地基。

## 12. 版本兼容、命令知识库与扩展策略

### 为什么必须考虑版本兼容

Minecraft 命令系统并不是完全静态的。
命令形式、参数能力、数据包结构、资源 ID 甚至部分语义都会随着版本变化。

因此，如果这个工具要认真支持：

- 原生命令编辑
- 参数发现
- 资源库输入
- 数据包导出

那么版本兼容就不能只在导出时临时补丁处理，而应成为产品模型的一部分。

### 版本兼容影响的核心范围

建议明确把版本影响拆成 4 类：

1. 命令能力
某条命令或某个参数是否存在、语义是否一致。

2. 资源数据
某些物品、方块、实体、效果、粒子、音效是否在目标版本中可用。

3. 数据包导出规则
命名空间、目录结构、`pack.mcmeta`、函数组织方式等是否符合目标版本要求。

4. 诊断与提示
某个工作流在当前项目版本下是否合法，是否只是“新版本才可用”。

只有这样拆开，后续产品提示才会准确。

### 项目级版本设定

建议每个项目都显式绑定一个目标 Minecraft 版本或版本档位。

例如：

1. 单一明确版本
如某个具体版本。

2. 兼容档位
如某一大版本范围或工具内部定义的规则集。

这个设定必须是项目元信息的一部分，而不是临时导出选项。

原因是：

1. 节点可用性取决于版本
2. 参数提示取决于版本
3. 资源库内容取决于版本
4. 导出结果取决于版本

### 命令知识库的定位

如果这个工具要让参数“可发现”，背后就需要一套命令知识库。

这套知识库不只是命令名称表，而应描述：

1. 命令定义
有哪些命令节点。

2. 参数结构
命令有哪些输入、哪些是必填、哪些是可选、哪些是高级参数。

3. 类型信息
每个参数对应什么数据类型。

4. 版本信息
在哪些版本可用、何时变更。

5. 示例与默认值
帮助面板和引导层展示。

也就是说，知识库是节点库、参数面板、资源提示和编译器之间的共同基础。

### 知识库应如何使用

建议前端体验尽量基于同一份命令知识库驱动：

1. 生成节点定义
决定节点显示什么输入输出。

2. 生成参数面板
决定显示哪些参数和控件。

3. 提供版本校验
判断当前版本下是否支持。

4. 提供默认值与示例
降低学习成本。

5. 支持错误提示
告诉用户是语义问题还是版本问题。

这样可以避免节点、面板、编译器各写一套规则，后续难以维护。

### 资源知识库的扩展策略

资源库同样不应只是静态列表。

建议资源知识库至少支持：

1. 资源类型划分
物品、方块、实体、粒子、音效等。

2. 多版本可用性标记
哪些资源在目标版本里存在。

3. 搜索别名
便于按常见名称或中文用途搜索。

4. 分类标签
支持资源面板筛选。

这样才能真正支撑“资源可发现性”和版本提示。

### 版本不兼容时的产品行为

建议工具不要简单粗暴地禁止一切，而应分层处理：

1. 可直接阻止的情况
例如目标版本根本不支持该命令或导出结构无法成立。

2. 可警告但允许继续的情况
例如资源可能在低版本中不可用，或某参数存在兼容风险。

3. 可自动降级的情况
如果某些表现层特性可以回退到旧规则，可提示用户并允许转换。

这比统一弹“版本不支持”更符合实际使用场景。

### 扩展策略的基本原则

既然命令知识库和资源知识库会持续扩展，必须提前定义扩展边界。

建议遵循这 5 条原则：

1. 新节点优先通过知识库扩展，而不是硬编码页面逻辑

2. 新参数优先通过配置驱动面板生成，而不是重复手写表单

3. 版本差异优先通过规则数据处理，而不是散落条件分支

4. 资源扩充优先通过数据源更新，而不是逐个组件补丁

5. 自定义函数节点应建立在稳定基础节点和类型体系之上

这样后续扩展成本才可控。

### 对自定义函数节点的影响

这一层设计也会反过来影响自定义函数节点。

因为如果基础节点和接口类型本身受版本影响，那么函数节点也必须具备：

1. 版本归属
它在哪个项目版本下定义。

2. 接口稳定性
如果内部引用了某个版本特有命令，外部需要可诊断。

3. 导出兼容性
函数节点映射到数据包导出时，必须符合目标版本规则。

这再次说明版本兼容不是外围问题，而是贯穿整个产品的基础能力。

### 最小可行版本建议

这一层的 MVP 不必一开始支持非常细粒度的全版本差异。

建议最小版本先做到：

1. 项目级版本设定

2. 节点和资源的基本版本可用性判断

3. 参数面板能根据版本隐藏或标记不兼容项

4. 导出阶段能给出明确版本相关错误或警告

先建立稳定框架，再逐步扩充知识库深度。

### 这一节的结论

如果没有版本兼容与知识库策略，前面的：

- 参数可发现性
- 资源库
- 导出能力
- 自定义函数节点

都只能停留在“看起来成立”，而不是“真正可靠”。

所以这部分虽然偏基础设施，但对于一个面向 Minecraft 命令系统的低代码工具来说，是不可省略的一层设计。

## 13. 协作、分享与工作流交换

### 为什么这一层重要

这类工具天然具备“作品可交换”的属性。

用户不仅会自己搭建工作流，还很可能会：

1. 把一段命令逻辑分享给别人
2. 复用别人做好的函数节点或子图
3. 交换项目模板
4. 在社区中积累常见解决方案

如果产品只支持“本地单人编辑当前画布”，那它的成长空间会被明显限制。

### 分享对象的层次

建议把可交换内容分成 4 层：

1. 单个工作流
一张完整的节点图。

2. 子图 / 函数节点
一段被封装好的可复用逻辑。

3. 项目
包含主工作流、子图、资源配置和导出配置。

4. 模板或示例
用于教学、起步或社区沉淀的标准化内容。

这样不同复杂度的分享需求都能有对应出口。

### 分享的最小单位建议

从产品实用性看，建议优先支持这两类交换：

1. 项目导入导出
用于完整保存和跨设备迁移。

2. 函数节点 / 子图导入导出
用于复用和社区共享。

这两类价值最高，也最符合你前面提出的“封装与扩展”方向。

### 项目分享的基本要求

如果用户要把一个项目发给别人继续编辑，建议至少保证：

1. 主工作流完整

2. 所有关联子图完整

3. 资源引用可还原

4. 目标版本信息保留

5. 导出配置一并带走

否则别人打开时就会出现“图能看见，但跑不起来”的问题。

### 函数节点分享的基本要求

函数节点是非常适合社区流通的内容，但前提是它本身足够自描述。

建议每个可导出的函数节点至少包含：

1. 名称

2. 描述

3. 输入输出接口定义

4. 内部子图

5. 版本适配信息

6. 所依赖的资源或前置要求说明

这样别人导入时，才不会只看到一个“黑盒节点”。

### 模板系统的定位

模板不应与“项目”或“函数节点”混淆。

建议这样理解：

1. 项目
是用户的真实工作成果。

2. 函数节点
是可复用逻辑模块。

3. 模板
是可作为起点的半成品或标准范式。

模板特别适合承载：

1. 新手示例
2. 常见命令逻辑
3. 数据包结构起步工程
4. 特定玩法场景的基础骨架

### 协作不一定等于实时多人

这类工具的协作价值不一定首先来自实时多人编辑。

更现实、收益更高的第一步通常是：

1. 可导出项目文件
2. 可导入项目文件
3. 可交换函数节点包
4. 可分享模板

也就是“异步协作”优先，而不是一开始就做复杂的实时协同。

这符合 KISS，也更符合当前产品阶段。

### 导入时的产品行为

为了让交换体验可靠，导入阶段建议加入这些校验：

1. 版本兼容检查

2. 缺失资源或不兼容节点提示

3. 函数节点接口完整性检查

4. 名称冲突处理
例如导入同名函数节点时如何处理。

5. 安全边界
如果未来支持外部模板或社区内容，至少要防止结构损坏或非法数据导致编辑器崩溃。

### 分享体验建议

从产品体验上，建议分享动作尽量明确且低摩擦：

1. 导出项目
得到完整工作流文件。

2. 导出函数节点
得到可复用模块文件。

3. 另存为模板
把当前内容沉淀为起点资产。

4. 从模板创建项目
快速进入编辑。

用户应清楚知道自己分享的是“整个项目”还是“一段逻辑模块”。

### 与资源库和函数节点的关系

这一层会直接影响前面两项能力：

1. 如果函数节点可以分享，就必须具备稳定接口和版本信息

2. 如果项目可以分享，资源引用就不能只是本地临时状态

3. 如果模板可以交换，知识库和项目结构就必须足够稳定

也就是说，协作与分享不是独立能力，而是对基础模型的一次压力测试。

### 最小可行版本建议

这一层的 MVP 建议先做：

1. 项目导入导出

2. 子图 / 函数节点导入导出

3. 内置模板与示例项目分离

4. 导入时给出版本和依赖校验提示

先打通“异步交换”，再考虑更复杂的社区化能力。

### 这一节的结论

如果这个工具未来想真正形成用户积累和社区价值，那么：

- 项目要能交换
- 函数节点要能复用
- 模板要能沉淀

而这一切都建立在前面已经定义的项目模型、函数封装、资源模型和版本策略之上。

所以协作与分享虽然不是第一阶段必做，但必须在设计阶段提前预留。

## 14. 调试、检查与可观测性设计

### 为什么这一层不可缺

当工作流开始变复杂时，用户遇到的问题就不再只是“参数填错了”，而会变成：

1. 为什么这段逻辑没有执行
2. 为什么结果和预期不一致
3. 为什么某个节点看起来连上了却没生效
4. 为什么导出成功但游戏里行为不对

如果工具只能“编译成功 / 编译失败”，用户最终还是会退回到原版里手动排查。

因此，对这种可视化工作流工具来说，调试与检查能力不是高级附加项，而是基本可用性的组成部分。

### 三层问题发现机制

建议把问题发现机制设计成三层：

1. 即时层
用户编辑节点和连接时，立刻发现明显错误。

2. 工作流检查层
用户主动触发检查或切换结果面板时，获得结构级诊断。

3. 编译 / 导出层
在真正生成结果时，发现更深层的组合问题、版本问题或导出问题。

这样用户不会把所有问题都推迟到最后一步才看到。

### 即时层应暴露的问题

建议即时层至少覆盖：

1. 必填参数缺失

2. 类型不兼容连接

3. 节点未接入主流程

4. 接口不完整的函数节点

5. 当前节点存在明显非法配置

这类问题应尽量在节点层直接标识，而不是只出现在结果区。

### 工作流检查器的定位

建议引入一个明确的“工作流检查器”概念，而不是把所有诊断都塞进编译结果。

这个检查器负责从结构角度评估整张图，而不是只关注最终文本输出。

它可以检查：

1. 是否存在多个孤立片段

2. 主流程是否有明确入口和出口

3. 是否存在未使用节点

4. 是否存在循环风险或非法回路

5. 封装函数接口是否与内部实现一致

6. 是否存在版本不兼容项

这会显著提升大图的可维护性。

### 节点级可观测性

建议每个节点都应具备可观察状态，而不只是“被选中/未选中”。

至少应包含：

1. 正常

2. 缺参

3. 警告

4. 错误

5. 未接入

6. 已封装或引用外部子图

这样用户在画布上扫一眼，就能发现问题热点。

### 连线级可观测性

很多问题并不发生在节点本身，而发生在连接关系上。

因此建议连线也应支持状态表达：

1. 正常连接

2. 类型风险

3. 条件路径

4. 已失效接口引用

5. 指向版本不兼容输出

连线不应永远只是静态视觉元素，而应成为可诊断对象。

### 调试视图建议

如果后续产品要支持更复杂工作流，建议预留“调试视图”或“检查视图”能力。

它不一定要一开始就做到运行时模拟，但至少可以提供：

1. 主流程高亮

2. 问题节点筛选

3. 未接入节点筛选

4. 函数节点展开关系浏览

5. 输入来源追踪

这能显著降低大图排查成本。

### 输入来源追踪

在参考 ComfyUI 的产品里，用户很关心“这个值到底从哪来的”。

所以建议支持：

1. 从参数字段反查输入来源节点

2. 从资源节点追踪其影响的下游节点

3. 从结果区反查参与生成的关键节点

如果没有这类追踪能力，复杂工作流一旦出问题就很难排。

### 函数节点调试要求

既然支持子图封装和函数节点，那么调试能力必须跨越边界。

至少要支持：

1. 从外部函数节点跳入内部子图

2. 从内部子图返回外层上下文

3. 看到函数节点接口与内部实际连接是否一致

4. 标识函数节点内部错误是否向外层透出

否则一旦函数节点变成黑盒，复用越多越难排错。

### 结果与诊断联动

编译反馈区和调试能力不应分裂。

建议做到：

1. 在结果区点击问题项时，画布自动聚焦

2. 在节点上点击错误标识时，结果区跳到对应说明

3. 在调试视图中筛选问题时，结果区同步更新

这会让“诊断”成为一个连续工作流，而不是两个孤立区域。

### 当前工具潜在的风险点

基于前面的设计目标，未来若不补这一层，最容易出现的问题是：

1. 节点越来越多，但排错效率没有同步提升

2. 函数节点越封装越黑盒

3. 用户能搭出图，但无法理解为什么结果不对

4. 编译成功被误解为逻辑正确

这会直接伤害工具可信度。

### 最小可行版本建议

这一层的 MVP 建议先做：

1. 节点级状态标识

2. 工作流检查器

3. 结果区与画布的问题跳转

4. 输入来源高亮

5. 函数节点内外跳转

先把“能排查”建立起来，再考虑更高级的模拟或执行追踪。

### 这一节的结论

对这种节点式低代码工具来说：

- 能搭建，只解决了一半问题
- 能理解、能排查、能修复，才算真正可用

所以调试、检查与可观测性设计，应被视为产品核心能力，而不是工程细节。

## 15. 设计收束与实施优先级重排

### 为什么需要重新收束

随着设计稿逐步展开，产品已经不再只是一个“节点编辑页面”，而是明确演化成一个包含：

- 节点工作台
- 参数与资源输入体系
- 自定义函数封装
- 版本与知识库
- 项目模型
- 分享交换
- 调试检查

的完整低代码环境。

在这种情况下，如果不重新做一次优先级收束，后续实施会很容易陷入“每个点都重要，结果没有核心推进顺序”的问题。

因此，这一节的目标不是增加新设计，而是明确：
哪些能力是第一优先级，哪些能力必须预留但可以后做，哪些能力是建立在前面之上的扩展层。

### 产品能力的六个层级

基于当前设计稿，建议把产品能力分成六层：

1. 核心闭环
用户从添加节点、配置参数、连接逻辑，到生成结果的最短可用链路。

2. 结构能力
让工作流在规模增长时仍然可读、可维护，包括函数封装和子图。

3. 资源能力
让参数输入从手写 ID 升级为可搜索、可复用、可连接的资源体系。

4. 版本能力
让命令、资源和导出结果具备真实可用性，而不是只在当前环境下“看起来对”。

5. 协作能力
让项目、子图、模板和函数节点可以交换、沉淀和复用。

6. 调试能力
让用户在图变复杂后，仍能理解、定位和修复问题。

这六层不是并列启动的，而应有严格先后关系。

### 第一优先级：核心闭环

第一优先级必须始终是：

1. 节点库可用
2. 画布连接逻辑清晰
3. 参数面板可高效编辑
4. 编译结果可读
5. 基础文案无乱码

理由很简单：
如果这条链路本身还不成立，后面所有高级能力都没有落点。

这一层的目标不是“功能多”，而是“第一次使用就能顺利完成一个工作流”。

### 第二优先级：结构能力

一旦核心闭环成立，下一优先级应是结构能力，而不是立刻扩大量节点。

也就是：

1. 子图折叠
2. 自定义函数节点
3. 主图与子图统一模型

原因是：
没有结构能力，用户一旦开始认真使用，画布复杂度会迅速失控。

这层能力是从“可用”走向“可持续使用”的关键拐点。

### 第三优先级：资源能力

资源能力应紧随结构能力之后。

包括：

1. 资源库
2. 资源搜索与分类
3. 面板选择与资源节点双通道
4. 常用资源与最近使用

原因是：
用户的高频痛点之一就是手写资源 ID 和参数不直观。
这层能力能显著提升低代码体验的实际价值。

### 第四优先级：调试能力

调试能力应尽早进入，但从实施排序看，建议在结构能力和资源能力基本成立后重点投入。

包括：

1. 节点状态标识
2. 工作流检查器
3. 结果与画布联动跳转
4. 输入来源追踪
5. 函数节点跨层调试

原因是：
只有当图开始变复杂时，调试能力的价值才全面显现。
但如果等到很晚再做，早期模型可能已经不利于调试设计。

所以这层在实施上应“早预留、后完善”。

### 第五优先级：版本能力

版本能力在用户感知上可能不是最先看到的，但在系统设计上必须早预留。

建议策略是：

1. 模型先支持
项目级版本、节点可用性、资源可用性先建模。

2. 体验后补强
逐步完善版本警告、降级策略和知识库深度。

这意味着：
版本能力不一定第一时间全面做完，但绝不能在数据模型上缺位。

### 第六优先级：协作与分享能力

协作与分享的重要性很高，但它建立在前面几层已经稳定的前提上。

建议顺序是：

1. 先有稳定项目结构
2. 再有稳定函数节点模型
3. 再做项目交换与模板沉淀

否则导入导出只会把不稳定结构扩大化。

因此这一层更适合作为“产品开始形成资产沉淀”后的增强项。

### 哪些能力必须现在就预留

虽然不是所有能力都要第一阶段实现，但有些能力必须从现在就进入模型设计：

1. 主图与子图统一为工作流

2. 节点数据中区分配置值与连线来源

3. 项目级版本信息

4. 资源独立模型

5. 函数节点接口定义

这些一旦后补，重构代价会非常高。

### 哪些能力可以延后但不能忽略

可以后做，但必须在设计上已经明确边界的有：

1. 函数节点库共享

2. 模板市场或社区内容

3. 更高级的资源知识库

4. 更细粒度的版本兼容规则

5. 更强的调试视图和模拟能力

这类内容适合作为第二轮、第三轮增强项。

### 建议的实施总顺序

基于整份设计稿，建议总顺序收敛为：

1. 修复可读性问题与基础文案

2. 打稳核心工作台闭环

3. 重构节点、参数、连线的基础语义

4. 建立主图 / 子图统一模型

5. 引入自定义函数节点与封装能力

6. 建立资源库与输入节点体系

7. 建立工作流检查器与基础调试能力

8. 补强版本知识库与兼容提示

9. 打通项目、函数节点和模板的导入导出

这个顺序既符合用户价值，也符合工程可控性。

### 这份设计稿的最终收束

到这里，这个产品的设计目标已经可以明确概括为：

它不是一个简单的 Minecraft 命令生成器，
也不是一个泛用流程图工具，
而是一个以 Minecraft 原生命令体系为核心语义、以蓝图式逻辑骨架和 ComfyUI 式参数体验为交互原则、支持资源输入、函数封装、版本兼容、调试检查与工作流复用的低代码工作台。

### 最终结论

如果后续实施严格围绕这份收束后的优先级推进，那么产品会按下面路径成长：

1. 先成为一个真正好用的命令可视化编辑器
2. 再成为一个能承载复杂逻辑的工作流工具
3. 最后成为一个可积累、可复用、可交换的 Minecraft 低代码编排平台

这应当是当前阶段最合理、也最稳的产品演进方向。
